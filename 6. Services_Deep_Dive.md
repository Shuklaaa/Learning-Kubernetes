# Why Pods Alone Are Not Enough

Every Pod gets its own IP.

Example:

```
Pod-1
10.244.0.5

Pod-2
10.244.0.6
```

If Pod-1 crashes,

```
Pod-1 deleted

↓

New Pod

↓

10.244.0.21
```

The application trying to communicate using **10.244.0.5** will fail.

Pods are **ephemeral**.

Therefore we need a stable endpoint.

That stable endpoint is a **Service**.

# Step 3: Create a Service

Example:

```yaml
selector:
  app: sample-python-app
```

The selector must match the labels on the Pods.

Pods:

```yaml
labels:
  app: sample-python-app
```

Service:

```yaml
selector:
  app: sample-python-app
```

If they don't match,

```
Service

↓

No Pods Found

↓

No Endpoints

↓

Application Not Reachable
```

# Service Discovery

A Service never searches Pods using:

* Pod name ❌
* Pod IP ❌

Instead it searches using **labels**.

```
Service

↓

Selector

↓

Matching Pods
```

This is called **Service Discovery**.

```

```

↓

↓

Pod

↓

# Understanding the Ports

The video spends time explaining these three ports.

```

↓

↓

```

Meaning:

* **NodePort** is exposed externally.
* **Port** is the Service port.
* **TargetPort** is where the container is actually listening.

# Important Interview Points

### Why do we need Services?

Because Pod IPs change whenever Pods are recreated.

---

### How does a Service find Pods?

Using **labels** and **selectors**.

---

### What happens if labels and selectors don't match?

The Service has no endpoints and cannot forward traffic.

---

### Difference between ClusterIP and NodePort

| ClusterIP              | NodePort                            |
| ---------------------- | ----------------------------------- |
| Internal only          | Internal + External                 |
| Default Service Type   | Exposes application outside cluster |
| Reachable only by Pods | Reachable by browser using Node IP  |

---

### What is an Endpoint?

Endpoints are the list of Pod IPs that a Service forwards traffic to.

---

### What is Kubeshark?

Kubeshark is a network observability tool for Kubernetes that captures and visualizes traffic between Pods.

---

# Key Takeaways

* Pods have dynamic IP addresses.
* Services provide a stable endpoint.
* Services discover Pods using labels and selectors.
* Endpoints contain the actual Pod IPs backing a Service.
* NodePort exposes the application outside the cluster.
* ClusterIP allows communication only within the cluster.
* Verifying the application inside the Pod using `curl localhost:8000/demo` helps isolate application issues before testing Service connectivity.
* Kubeshark provides a visual representation of how requests travel from the client to the Service and finally to the Pods.

# Important Point About NodePort

Abhishek explains that NodePort **does not expose your application to the entire Internet.**

It is generally reachable only by machines that can access the Kubernetes node.

For example, if your laptop has IP:

```
192.168.49.2
```

then another machine on the **same network** can access:

```
http://192.168.49.2:30007/demo
```

provided there are no firewall or routing restrictions.

However, someone on the public Internet **cannot** access this NodePort directly.

# Why Do We Need a LoadBalancer?

Suppose you deploy your application on a cloud platform like AWS.

```
Internet

↓

Public IP (Load Balancer)

↓

NodePort

↓

ClusterIP Service

↓

Pods
```

A **LoadBalancer Service** creates (or integrates with) a cloud load balancer that gets a **public IP address**.

The load balancer receives traffic from anywhere on the Internet and forwards it to the Kubernetes nodes, which then forward it to the appropriate Pods.

Therefore:

* **ClusterIP** → Only inside the cluster.
* **NodePort** → Reachable through the Kubernetes node's IP (typically within the same network).
* **LoadBalancer** → Publicly accessible from the Internet (using a cloud provider's load balancer).
