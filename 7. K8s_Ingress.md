# What is Ingress?

**Ingress** is a Kubernetes API object that manages **external HTTP/HTTPS access** to services running inside a Kubernetes cluster.

Instead of exposing every application using separate **NodePort** or **LoadBalancer** services, Ingress provides **a single entry point** and routes requests to the appropriate service based on defined rules.

Think of **Ingress as a reverse proxy or traffic manager** for your Kubernetes cluster.

---

# Why Do We Need Ingress?

Before Ingress, applications were exposed using:

- ClusterIP
- NodePort
- LoadBalancer

These methods work but have some limitations.

## Problem 1: Too Many LoadBalancers

Suppose you have four applications:

- User Service
- Product Service
- Payment Service
- Order Service

Without Ingress:

```text
User Service ------> LoadBalancer (Public IP)

Product Service ---> LoadBalancer (Public IP)

Payment Service ---> LoadBalancer (Public IP)

Order Service -----> LoadBalancer (Public IP)
```

Each LoadBalancer gets its own public IP.

### Problems

- Higher cloud cost
- Difficult to manage
- More public IP addresses

---

## Problem 2: Limited Routing

A Kubernetes Service can:

- Discover Pods
- Perform basic load balancing

But it **cannot** perform advanced routing like:

- Host-based routing
- Path-based routing
- SSL/TLS termination
- URL rewriting
- Authentication
- Rate limiting
- IP whitelisting

---

# Solution: Ingress

Ingress exposes only **one public endpoint**.

```text
                Internet
                    |
                    |
          +-------------------+
          | Ingress Controller|
          +-------------------+
           /        |        \
          /         |         \
         /          |          \
    User       Product      Payment
    Service      Service      Service
```

Benefits:

- One public IP
- Lower cost
- Centralized routing
- Easier management
- Secure HTTPS support

# What Can Ingress Do?

## 1. Host-Based Routing

Different domains route to different services.

Example:

```text
shop.example.com
        |
        v
 Shop Service
```

```text
admin.example.com
        |
        v
 Admin Service
```

---

## 2. Path-Based Routing

One domain can serve multiple applications.

Example:

```text
example.com/products
        |
        v
 Product Service
```

```text
example.com/orders
        |
        v
 Order Service
```

```text
example.com/users
        |
        v
 User Service
```

---

## 3. HTTPS / TLS

Ingress can terminate SSL certificates.

```text
Internet (HTTPS)
        |
        v
Ingress Controller
        |
        v
HTTP Inside Cluster
```

Clients communicate securely while backend services can still use HTTP.

---

# Ingress vs Ingress Controller

Many beginners confuse these two.

## Ingress

- Kubernetes resource
- YAML file
- Contains routing rules
- Does **not** actually route traffic

Example:

```yaml
kind: Ingress
```

---

## Ingress Controller

Ingress Controller is the software that reads the Ingress resource and configures the reverse proxy.

Examples:

- NGINX
- Traefik
- HAProxy
- Ambassador

Without an Ingress Controller:

```text
Ingress YAML
     |
     X
Nothing Happens
```

The Ingress resource is useless unless an Ingress Controller is installed.

# Example Ingress YAML

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress

metadata:
  name: my-ingress

spec:
  ingressClassName: nginx

  rules:
    - host: foo.bar.com

      http:
        paths:
          - path: /bar
            pathType: Prefix

            backend:
              service:
                name: service1
                port:
                  number: 80
```

---

# Understanding the YAML

## ingressClassName

```yaml
ingressClassName: nginx
```

Specifies which Ingress Controller should process this resource.

---

## Host

```yaml
host: foo.bar.com
```

Requests for this hostname will match this rule.

---

## Path

```yaml
path: /bar
```

Only requests starting with `/bar` are routed.

---

## Backend

```yaml
backend:
  service:
    name: service1
    port:
      number: 80
```

Traffic is forwarded to:

- Service Name
- Service Port

The Service then forwards traffic to Pods.

# Ingress vs LoadBalancer

| LoadBalancer | Ingress |
|--------------|---------|
| One Public IP per Service | One Public IP for Multiple Services |
| Expensive | Cost Effective |
| Basic Routing | Advanced Routing |
| No Host-Based Routing | Host-Based Routing |
| No Path Routing | Path-Based Routing |
| No TLS Termination | TLS Termination |
| Simple Exposure | Centralized Traffic Management |

# Interview Questions

## What is Ingress?

Ingress is a Kubernetes resource that manages external HTTP/HTTPS access to services using routing rules.

---

## Why do we need Ingress?

- Reduces cloud cost
- One public IP
- Host-based routing
- Path-based routing
- HTTPS support
- Centralized traffic management

---

## What is an Ingress Controller?

An Ingress Controller is software that watches Ingress resources and implements the routing rules.

---

## Can Ingress work without an Ingress Controller?

**No.**

Ingress is only a configuration object.

Without an Ingress Controller, Kubernetes will ignore the Ingress resource.

---

# Key Takeaways

- Ingress is used to expose HTTP/HTTPS services.
- It provides one external entry point for multiple applications.
- It supports host-based and path-based routing.
- It enables TLS termination.
- An **Ingress resource** only defines routing rules.
- An **Ingress Controller** actually processes those rules.
- NGINX is the most popular Ingress Controller.
- On Minikube, enable the Ingress addon before creating Ingress resources.
